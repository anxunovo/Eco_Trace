# 校园二手漂流 Web 端需求文档（MVP 版）

**项目暂定名：碳循校园：校园二手漂流与碳减排估算平台**  
**文档用途：交给 Codex / 开发工具直接实现 Web 端 Demo**  
**版本：v0.1 / MVP Demo**  
**项目类型：科技作品类，软件 / 设计 / 现场演示**

---

## 1. 项目定位

本项目是一个面向中国校园场景的二手漂流 Web 平台。平台允许学生发布闲置物品、剩余食品或可再利用资源，支持自由标价、免费赠送和物品交换。用户上传图片和描述后，系统通过 AI 辅助识别物品类型，并估算物品成功流转后可能节约的碳排放量。

平台核心不是电商交易，而是校园闲置资源再利用。节能减排比赛中的表达重点是：每一次闲置物品成功流转，都代表一次资源浪费的减少；系统把这种再利用行为转化为可视化的碳减排贡献。

### 1.1 一句话说明

让校园里的闲置物品、剩余食品和可再利用资源更容易被看见、更容易被带走、更容易被再次使用，并把每一次成功流转转化为可展示的节能减排成果。

### 1.2 产品基调

- 校园场景优先，不做通用电商平台。
- 主打闲置物品再利用，比赛表达侧重节能减排。
- 支持付费、免费、交换、面议等灵活方式。
- 平台只做信息撮合，不接入支付，不托管资金，不做担保交易。
- AI 负责辅助识别、辅助填写、辅助估算减碳，不替代用户责任。
- 碳减排数据为估算值，用于环保贡献展示和比赛 Demo，不作为精确碳审计数据。

## 2. MVP 目标与非目标

### 2.1 MVP 目标

MVP 需要跑通以下完整链路：

1. 用户进入首页，看到平台定位、校园资源流转数据和物品列表。
2. 用户发布闲置物品或食物分享信息，上传图片并填写基础信息。
3. 系统调用 AI 识别逻辑，自动生成物品类别、重量估算、标题建议、描述建议和预计碳减排值。
4. 发布者可以手动修改 AI 结果中的类别、重量、交易方式、价格、地点、取物方式等信息。
5. 其他用户可以浏览、筛选、搜索物品，并查看物品详情。
6. 用户可以点击“我想要 / 联系发布者”，弹出联系方式或站内留言入口。
7. 双方线下自行完成付费、赠送或交换。
8. 发布者点击“确认已流转”，平台将物品状态改为已完成，并把预计减碳值计入个人和校园总贡献。
9. 食物类物品具有保质期 / 最晚领取时间，过期后自动标记为已过期，不再计入可领取列表。
10. 首页或数据看板展示累计流转次数、累计估算减碳、累计减少食物浪费等 Demo 指标。

### 2.2 非目标

MVP 不实现以下内容：

- 不接入微信支付、支付宝或银行卡支付。
- 不做平台担保交易、退款、仲裁或纠纷处理。
- 不做物流系统。
- 不做真实学籍认证，Demo 可使用模拟登录或校内身份展示字段。
- 不做复杂即时聊天，MVP 可用联系方式弹窗或简单留言替代。
- 不承诺碳减排数据的科学审计准确性。
- 不对食品安全提供担保，只做信息提示与风险边界展示。

## 3. 用户角色

| 角色 | 说明 | MVP 权限 |
|---|---|---|
| 游客 | 未登录访问者 | 浏览首页、列表、详情，不能发布和确认流转 |
| 学生用户 | 默认登录用户，可用模拟身份 | 发布物品、编辑自己的发布、联系发布者、确认自己的物品已流转 |
| 发布者 | 发布了物品的学生用户 | 管理自己的发布，修改状态，确认完成 |
| 需求方 / 认领者 | 想购买、免费领取或交换的人 | 查看详情，点击联系，发起“我想要” |
| 管理员 | Demo 管理端角色，可选 | 查看所有发布、下架违规内容、维护分类和碳系数 |
| AI 服务 | 系统内部服务 | 图片识别、字段建议、碳减排估算 |

MVP 可以先使用固定模拟账号，不强制实现完整注册登录。建议至少提供“当前用户切换”能力，方便演示发布者和认领者不同身份。

## 4. 核心业务规则

### 4.1 交易方式

发布者在发布时必须选择一种流转方式：

| 枚举值 | 页面展示 | 说明 |
|---|---|---|
| `SALE` | 付费转让 | 发布者自由标价，平台不处理支付 |
| `FREE` | 免费赠送 | 0 元，适合急出、毕业季、宿舍清理 |
| `SWAP` | 物品交换 | 发布者填写希望交换的物品或条件 |
| `NEGOTIABLE` | 面议 / 自行协商 | 不明确标价，双方自行沟通 |

### 4.2 支付边界

平台允许付费信息展示，但不做支付。物品详情页不能出现“立即支付”“下单”“付款”等按钮。建议按钮文案为：

- 我想要
- 联系发布者
- 约定交接
- 确认已流转

详情页需要展示提示：

> 平台仅提供校园闲置信息撮合，不接入支付，不托管资金。付费、交换或赠送由双方自行协商完成。

### 4.3 完成流转规则

MVP 阶段采用“发布者确认完成”为主：

1. 物品发布后状态为 `ACTIVE`。
2. 有人点击“我想要”后，状态仍为 `ACTIVE`，但记录一次意向。
3. 发布者线下完成交易、赠送或交换后，点击“确认已流转”。
4. 系统将状态改为 `COMPLETED`。
5. 系统生成一条 `CarbonRecord`，把该物品的估算减碳值计入总数据。

后续版本可以升级为双方确认、扫码确认或积分激励确认。

### 4.4 成功流转与减碳关系

- 未完成流转时，只显示“预计可节约”。
- 完成流转后，显示“已节约”。
- 只有 `COMPLETED` 状态的物品计入总减碳数据。
- `EXPIRED`、`REMOVED`、`DRAFT` 状态不计入总减碳。

### 4.5 食物类规则

食物类物品必须放入 MVP，但需要单独处理安全边界。

食物发布必须填写：

- 食物类型。
- 是否未开封 / 已开封 / 现制食品 / 活动剩余食品。
- 估算重量或份数。
- 保质期或最晚领取时间。
- 领取地点。
- 发布者补充说明。

食物详情页必须展示风险提示：

> 食物信息由发布者自行填写，平台仅提供信息展示、撮合和减碳估算，不对食品质量和食品安全作担保。领取前请自行确认包装、保质期、储存条件和实际状态。

食物状态规则：

- 超过最晚领取时间后，系统自动标记为 `EXPIRED`。
- 已过期食物不显示在默认列表中。
- 已过期食物不能被确认流转。
- 食物类碳减排按食物类型和重量估算。

## 5. Web 端页面需求

### 5.1 首页 `/`

目标：让评委或用户一眼理解项目价值。

首页模块：

1. 顶部导航：Logo、首页、发现闲置、发布物品、减碳看板、关于项目。
2. Hero 区：项目 slogan、核心说明、主要 CTA。
3. 校园累计数据：累计流转件数、累计估算减碳 kg CO₂e、累计食物减少浪费 kg、参与学生数。
4. 分类入口：教材资料、电子产品、宿舍用品、衣物鞋包、文具工具、运动用品、食物分享、免费专区、交换专区、毕业季专区。
5. 最新发布物品卡片列表。
6. 减碳机制说明：上传图片、AI 识别、估算减碳、完成流转、记录贡献。
7. 底部说明：平台不接入支付，仅做信息撮合。

首页 CTA：

- 发布我的闲置
- 浏览校园闲置
- 查看减碳成果

### 5.2 发现闲置页 `/listings`

目标：浏览和筛选校园闲置物品。

筛选项：

- 关键词搜索。
- 分类筛选。
- 流转方式：付费、免费、交换、面议。
- 价格区间。
- 校区 / 地点。
- 状态：可领取、已完成、已过期。
- 排序：最新、预计减碳最高、免费优先、临近过期食物优先。

物品卡片字段：

- 主图。
- 标题。
- 分类标签。
- 流转方式标签。
- 价格或交换说明。
- 交接地点。
- 新旧程度。
- 预计减碳值。
- 食物类显示最晚领取时间。
- 状态标签。

### 5.3 物品详情页 `/listings/:id`

目标：展示完整信息并完成撮合动作。

详情页字段：

- 图片轮播。
- 标题。
- 价格 / 免费 / 交换 / 面议信息。
- 分类。
- 物品状态。
- 新旧程度。
- 发布者昵称和校区。
- 发布时间。
- 交接地点。
- 描述。
- AI 识别结果。
- 预计减碳值及估算说明。
- 联系发布者按钮。
- 我想要按钮。
- 发布者本人可见：编辑、下架、确认已流转。

详情页提示：

- 碳减排为估算值，仅用于环保贡献展示。
- 平台不接入支付，不承担交易担保。
- 食物类额外展示食品安全提示。

### 5.4 发布页 `/publish`

目标：快速发布，突出 AI 辅助。

建议采用分步表单：

#### Step 1：上传图片

- 上传 1-5 张图片。
- 主图必填。
- 上传后可点击“AI 识别并生成信息”。

#### Step 2：AI 识别结果

展示 AI 建议字段，允许用户修改：

- 建议标题。
- 识别分类。
- 估算重量。
- 新旧程度。
- 物品描述建议。
- 预计减碳值。
- 识别置信度。
- 估算依据说明。

#### Step 3：发布者补充信息

必填字段：

- 标题。
- 分类。
- 流转方式。
- 价格：付费转让时必填，可为 0。
- 交换需求：交换模式时必填。
- 描述。
- 校区 / 交接地点。
- 联系方式展示方式。
- 新旧程度。

食物类额外字段：

- 食物类型。
- 估算重量或份数。
- 包装状态。
- 生产 / 制作时间。
- 保质期或最晚领取时间。
- 储存条件。
- 食品安全确认勾选。

#### Step 4：预览并发布

展示物品卡片预览、详情页预览、预计减碳信息和平台边界提示。点击发布后进入详情页。

### 5.5 减碳看板 `/impact`

目标：把闲置流转成果转化为比赛展示数据。

模块：

- 校园累计估算减碳。
- 累计完成流转件数。
- 累计减少食物浪费重量。
- 分类减碳贡献柱状图。
- 最近完成流转记录。
- 个人贡献排行。
- 说明：估算方法、数据边界、非精确审计。

### 5.6 我的发布 `/me/listings`

功能：

- 查看我发布的物品。
- 状态筛选：草稿、发布中、已完成、已过期、已下架。
- 编辑。
- 下架。
- 确认已流转。
- 查看意向用户数量。

### 5.7 管理端 `/admin`（可选但建议实现简版）

功能：

- 查看全部发布。
- 按状态、分类、食物类筛选。

## 7. AI 识别与减碳估算需求

### 7.1 AI 识别输入

输入包含：

- 物品图片。
- 用户手动填写的标题或描述，可为空。
- 用户选择的分类，可为空。
- 食物类额外字段，可为空。

### 7.2 AI 识别输出 Schema

```json
{
  "titleSuggestion": "九成新高等数学教材",
  "category": "BOOKS",
  "isFood": false,
  "condition": "GOOD",
  "estimatedWeightKg": 0.8,
  "descriptionSuggestion": "适合低年级同学复习使用，书页完整，有少量笔记。",
  "carbonFactor": 1.3,
  "estimatedCarbonSavedKg": 0.7,
  "confidence": 0.78,
  "assumptions": [
    "根据图片判断为纸质教材",
    "按单本教材约0.8kg估算",
    "按二手再利用替代部分新购需求估算"
  ]
}
```

### 7.3 实现要求

Codex 实现时应把 AI 功能抽象成独立服务：

- `analyzeListingImage(input)`：负责图片识别和字段建议。
- `calculateCarbonEstimate(input)`：负责根据类别、重量、流转方式和食物信息计算碳减排。

MVP 可先使用 Mock AI，不必接入真实模型。Mock 规则：

1. 若用户已选分类，则优先使用用户分类。
2. 若没有真实图片识别能力，则根据标题关键词和分类生成建议。
3. 所有 AI 结果都必须允许用户修改。
4. AI 估算依据必须可展示，不要只给一个数字。
5. 所有碳数据保留一位小数，单位为 kg CO₂e。

### 7.4 默认演示系数

以下为 MVP 演示用系数，必须以配置形式保存，后续可替换为正式参考数据。

| 分类 | 计算方式 | 默认系数 | 默认重量 | 替代系数 | 说明 |
|---|---|---:|---:|---:|---|
| 教材书籍 | per_kg | 1.3 | 0.8 | 0.7 | 纸质资料再利用 |
| 衣物鞋包 | per_item | 8.0 | - | 0.7 | 单件衣物 / 鞋包估算 |
| 小型电子产品 | per_item | 25.0 | - | 0.6 | 耳机、台灯、小电器等 |
| 宿舍用品 | per_kg | 3.0 | 2.0 | 0.7 | 收纳、置物架、生活用品 |
| 文具工具 | per_item | 2.0 | - | 0.7 | 文具、小工具 |
| 运动用品 | per_item | 5.0 | - | 0.7 | 球拍、瑜伽垫等 |
| 普通食物 | per_kg | 2.5 | 0.5 | 1.0 | 按避免食物浪费估算 |
| 肉蛋奶类食物 | per_kg | 8.0 | 0.5 | 1.0 | 高碳食物估算项 |
| 果蔬类食物 | per_kg | 0.8 | 0.5 | 1.0 | 低碳食物估算项 |
| 零食饮料 | per_kg | 2.0 | 0.5 | 1.0 | 未开封零食、饮料等 |

计算公式建议：

```text
若 per_kg：
预计减碳 kgCO2e = 重量 kg × 分类碳系数 × 替代系数

若 per_item：
预计减碳 kgCO2e = 单件默认碳系数 × 替代系数
```

注意：

- 这是 Demo 估算，不作为权威碳核算。
- 页面展示必须写“预计”“估算”。
- 完成流转前不计入总减碳。

## 8. 状态流转

```text
DRAFT 草稿
  ↓ 发布
ACTIVE 发布中
  ↓ 发布者确认完成
COMPLETED 已流转

ACTIVE 发布中
  ↓ 食物超过最晚领取时间
EXPIRED 已过期

ACTIVE / DRAFT
  ↓ 发布者下架或管理员下架
REMOVED 已下架
```

状态规则：

- `DRAFT`：仅发布者可见。
- `ACTIVE`：列表默认显示。
- `COMPLETED`：详情可查看，列表可通过筛选查看，不再允许联系。
- `EXPIRED`：默认隐藏，食物类到期后自动产生。
- `REMOVED`：默认隐藏。

## 11. UI 风格要求

- 干净、轻量、校园感。

## 14. 验收标准

### 14.1 功能验收

- 可以打开首页并看到项目定位、分类入口、最新发布和累计数据。
- 可以进入列表页，完成关键词搜索、分类筛选、流转方式筛选。
- 可以进入详情页，看到物品信息、预计减碳、联系按钮和边界提示。
- 可以发布普通闲置物品。
- 可以发布食物类物品，并填写食物额外字段。
- 点击 AI 识别后，系统能返回字段建议和预计减碳值；即使是 Mock，也要表现出完整流程。
- 发布者可以确认已流转。
- 确认已流转后，物品状态变为已完成，平台累计减碳数据增加。
- 食物超过最晚领取时间后能标记为过期，默认不展示在可领取列表中。
- 不出现支付、下单、购物车、物流等功能。

### 14.2 展示验收

- 页面在桌面端和移动端都能正常显示。
- 物品卡片清晰展示价格 / 免费 / 交换状态。
- 碳减排数字在卡片、详情和看板中都可见。
- 食物类页面必须显示食品安全提示。
- 所有碳减排文案必须包含“预计”或“估算”。
- 平台边界必须明确：信息撮合，不接入支付。

### 14.3 Demo 验收路径

推荐演示路径：

1. 打开首页，说明项目用于校园闲置资源再利用。
2. 浏览最新发布，看到不同交易方式和预计减碳标签。
3. 进入食物分享物品详情，展示最晚领取时间和食品安全提示。
4. 发布一个闲置教材，上传图片，点击 AI 识别，得到分类和减碳估算。
5. 修改价格为免费或 10 元，选择交接地点，发布。
6. 进入我的发布，点击确认已流转。
7. 回到减碳看板，看到总减碳数据增加。

## 16. 给 Codex 的实现指令摘要

请根据本需求实现一个 Web 端 MVP。优先保证可演示、流程完整、页面清晰，不要实现支付系统。AI 识别可以先用 Mock 服务，但必须保留独立服务接口，方便后续替换为真实 AI。数据可以先使用本地 JSON、内存数据库或轻量数据库，并内置种子数据。页面需要突出校园场景、闲置再利用、自由流转方式和估算减碳贡献。

必须遵守：

- 不要出现购物车、订单支付、确认付款等电商闭环功能。
- 付费转让只展示价格和联系方式，不处理支付。
- 所有减碳数值都标注为预计或估算。
- 食物类必须显示食品安全边界提示。
- 发布者必须能够确认已流转，完成后才计入总减碳。
- 页面要适配桌面端和移动端。
